## Signature Flow
1. ผู้ใช้กด Export + toggle "ลายเซ็นดิจิทัล"
2. client: `const hash = await sha256Hex(title + content);`
3. call `createSignature({ runId?, signerName, signerPosition, agencyName, documentSubject, refNo, contentHash: hash })`
   → server insert `signed_documents` → return `{ id, signedAt }`
4. `verifyUrl = ${origin}/verify/${id}`
5. `qrDataUrl = await QRCode.toDataURL(verifyUrl)`
6. render PDF/DOCX with `signature` block

## Verify Page (`/verify/$id`)
- Public route
- เรียก `fetchVerifySignature(id)` (ใช้ anon RLS SELECT)
- แสดง: ✅ ใบลายเซ็นถูกต้อง / ❌ ไม่พบ
- รายละเอียด: signer_name, signer_position, agency_name, document_subject, ref_no, signed_at, content_hash (full)
- footer: อ้าง พ.ร.บ.ธุรกรรมทางอิเล็กทรอนิกส์ มาตรา 9

## ตราครุฑ (Letterhead)
- เก็บใน Supabase Storage bucket `agency-assets` (private)
- admin upload ผ่าน `/admin/settings`
- save `agency_settings.letterheadPath`
- server function `getLetterheadSignedUrl()` → return signed URL (TTL 5 นาที) ให้ client render
- ห้ามฝัง raw bytes ลงโค้ด

## ข้อห้าม
- อย่าฝังลายเซ็นภาพของผู้อื่น (ตรวจ `userId` กับ profile)
- อย่าทำ hash จาก content ที่ยังไม่ final (ต้องเป็น final text ที่จะ render)
- อย่าให้ verify page leak อย่างอื่นนอกจากฟิลด์ใน `VerifySignature`
